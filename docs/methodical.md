# Методические рекомендации по дисциплине "Конструирование интернет-приложений"

## 1. Общие сведения
Дисциплина включает:
- 8 демонстрационных занятий (лекции-практикумы) по 1,5 часа
- 8 лабораторных работ по 1,5 часа (с самостоятельным завершением)
- 8 занятий по курсовой работе (контрольные точки)

Общий результат: разработанное веб-приложение Service Desk с ролями, статусами, UI по Figma и поставкой через Docker/CI.

## 2. Требования к ПО и окружению
- Git, GitHub аккаунт
- Node.js + npm
- .NET SDK (LTS)
- Docker Desktop / Docker Engine
- PostgreSQL (через docker compose)
- IDE: Cursor (рекомендуется), VS Code/Visual Studio (по желанию)

## 3. Организация работы
### 3.1. Репозиторий
Студент работает в репозитории, полученном через GitHub Classroom.
Сдача каждой лабораторной: Pull Request с названием `LabXX`.

### 3.2. Figma
Figma используется как спецификация UI.
Ссылки на макеты и прототипы добавляются в README.

## 4. Требования к лабораторным работам
Каждая лабораторная:
- имеет цель и критерии приёмки
- сдаётся PR-ом
- защищается демонстрацией (UI + API + DB)

## 5. Курсовая работа
### 5.1. Содержание
Курсовая работа — итоговый проект, развивающий сквозной кейс.
Разрешены расширения: вложения, внутренние заметки, история изменений, отчёты.

### 5.2. Контрольные точки (8 занятий)
1) Figma и сценарии утверждены
2) ERD и эндпоинты согласованы
3) Backend MVP готов
4) Frontend skeleton + чтение данных
5) Формы + валидация
6) Auth + роли
7) Тесты + docker
8) Предзащита: демо + архитектура

## 6. Оценивание (рекомендуемая рубрика)
- Лабораторные (8) — 40%
  - выполнены критерии
  - качество кода и структуры
  - корректная работа в docker/CI
- Курсовая работа — 50%
  - полнота сценариев (MVP)
  - корректность ролей и безопасности
  - качество UX (состояния, ошибки, консистентность)
  - инженерные практики (миграции, тесты, README)
- Дисциплина разработки — 10%
  - регулярность коммитов, аккуратные PR
  - соблюдение дедлайнов

## 7. Чек-лист преподавателя на защиту
- проект запускается по README
- миграции применяются
- роли работают (401/403)
- Student не видит чужие тикеты
- Operator меняет статус по правилам
- UI имеет состояния loading/empty/error
- CI проходит

## 8. Типовые ошибки и рекомендации
- нет миграций в репо
- “работает только локально” без compose/README
- нет обработки ошибок UI
- смешение DTO и Entity
- отсутствие ограничений доступа к данным
